1\. 组网

对于海量数据的处理，需要 Hadoop 分布式计算，本项目中要处理的数据算不上“海量”，只是模拟这个过程。创建3台搭载 CentOs 7 操作系统的虚拟机，命名为 master + slave1 + slave2，内存分别为 4G、2G、2G。然后修改 Host 映射并配置静态IP。IP一般是动态的，如果不改成静态的，下次开机路由器会重新分配IP，Master 就找不到 slave 了。

关闭防火墙。在内网集群中，节点间互相通信的端口非常多，因此通常关闭内部防火墙，否则 DataNode 无法加入集群。

SSH 免密登录，因为 Master 节点需要频繁登录到 slave 节点去启动进程，所以要配置免密登录，基本原理在上一本笔记本上。

所有节点都安装上统一版本的 JDK，因为 Hadoop 是用 Java 写的，然后解压 Hadoop 压缩包，配置好环境变量。修改 Hadoop 目录下的几个核心 .xml 文件。之后在 Master 上建立文件系统目录树，然后执行  start-all.sh ，组网成功。

2\. 数据预处理

用 MapReduce，它是一种分布式计算的编程模型，字面意思是映射、归约。Map 阶段包括数据的清洗、过滤、提取，Reduce 阶段是汇总、聚合。MapReduce 清洗脚本是在 IDEA 中用 java 写的，编写了一个名叫  ChinaMapper  的 java 类，负责核心清洗业务，生成标准 CSV 数据。还有一个  ChinaDriver.java  主程序，作用是告诉集群数据从哪里读，洗完存在哪里。一开始 MapReduce 无法读取底层目录的子文件夹，要配置“递归读取”才能让任务吃到所有的数据。代码写完后，打成 jar 包给 Master 去运行，不过在这之前，Hadoop 集群得先把原始数据 .gz 压缩包解压了，才能提取原始数据，然后就可以开始清洗数据，大约用半小时。大数据有一条铁律“计算向数据移动”，因此不要动 Hadoop 集群上的原始数据，只需在 Windows 上编好程，把脚本传给 Hadoop。在 CentOS 7 上也能编程，但是没有 IDE，很麻烦，所以选择在 Windows 上编程传进去。

3\. 大数据分析和数据搬运

清洗好的标准 CSV 文件是存放在 Hive 数据仓库里的。经历了清洗、浓缩聚合后，现在我们的 Hive 仓库有 6 张核心的统计分析表，但 Web 前端无法直接读取它，因为它还是太大了。需要用到一个叫 Sqoop 的工具，把表批量导出到 MySQL。

4\. 模型训练

对于传统的多项式式模型，最高次4或5拟合结果较好，再高就会出现过拟合。但气温本身浮动就不大，如果仍采用多项式模型，必然导致过拟合。“时间序列模型”是专门用于分析气象的模型，它又可以分两种，一种叫指数平滑法，另一种叫 ARIMA 模型，这两种都要尝试，谁的预测精度最优就选谁。时间序列模型的硬性要求是：数据必须是等间距的绝对时间点，这是我们在数据预处理时应当遵循的要求。

“验证集划分”，不能把所有的数据都用来训练，要划分出一些数据作验证集，比如，可以用 00-21 年的数据当训练集，用 22 年的数据验证精度，如果精度高，那么可以认为这个模型预测 23 年的数据是准确的。但这和普通的 K折交叉验证有所不同，时间序列交叉验证要求不能用未来的数据训练过去的规律。

5\. Web 网页制作

用 Python 语言的 Django 框架搭建后端，HTML + ECharts 做前端，标准的 MVC 前后端分离模式。

首先做前端，用 HTML 写好盒子，规定长宽、位置，设置 ID。然后后端要从 MySQL 中拿数据，在  views.py  文件中使用  sqlalchemy  库远程连接到 CentOS7 虚拟机中的 MySQL 库，把要展示的数据提取到内存里。

\*\*注意：\*\*这里有一个问题，Python 拿到的是 Numpy 格式数据，而浏览器只认识 JSON 格式数据。如果直接传 Numpy 数据，浏览器脚本会崩溃。要专门写一个转化函数，把数据强行转化为标准 Python 字典，然后给前端传 JSON 数据。

ECharts 是百度开源的一个纯 JavaScript 图表库，它会接管 HTML 里面的盒子，要写一个配置字典规定各种参数，原本空荡荡的  <div>  就会被渲染成炫酷的交互式图表。
